♿ Our Commitment

Inclusive by Design

Accessibility at FinPay is not an afterthought — it's a core product value. We believe that financial tools should be available to every individual, and we invest in making that a reality.

♿
WCAG 2.1 Level AA Standard

FinPay targets conformance with the Web Content Accessibility Guidelines (WCAG) 2.1 at Level AA across all of our web products. We conduct regular audits with assistive technology users and automated testing tools to ensure we meet and exceed these standards.

🔍

Regular Audits

We run quarterly accessibility audits using both automated tools and manual testing with real assistive technology users.

🤝

Inclusive Testing

Our QA process includes team members and consultants who rely on screen readers, keyboard navigation, and other assistive tools daily.

🔄

Continuous Improvement

Every product sprint includes dedicated accessibility acceptance criteria. Regressions are treated as critical bugs and resolved with urgency.

⌨️ Keyboard Navigation

Navigate With Only a Keyboard

Every feature and workflow in FinPay is fully operable without a mouse. We follow a logical tab order, provide visible focus indicators, and support all standard keyboard interaction patterns.

ℹ️
Focus Indicators Are Always Visible

We never suppress the browser's native focus ring. Every interactive element displays a clear, high-contrast focus indicator that meets WCAG 2.1 AA focus appearance criteria.

🔊 Screen Reader Support

Works Seamlessly With Your Screen Reader

FinPay is designed and tested with leading screen readers to ensure every user can complete payment tasks independently and confidently.

🍎

VoiceOver (macOS & iOS)

Fully tested with Apple VoiceOver on both Safari and Chrome. All landmarks, headings, and live regions are announced correctly.

🪟

NVDA & JAWS (Windows)

Verified with NVDA and JAWS on Firefox and Chrome. Form instructions, validation errors, and status messages use ARIA live regions for real-time feedback.

🤖

TalkBack (Android)

Our mobile web experience is verified with Android TalkBack. Touch targets meet minimum size requirements and all interactive controls are properly labeled.

🏗️

Semantic HTML First

We build with native HTML elements wherever possible, using ARIA roles, properties, and states only to supplement where no semantic HTML equivalent exists.

✅
Dynamic Content Announcements

Form errors, success states, notifications, and loading indicators are all communicated through ARIA live regions and alert roles — so screen reader users never miss a status update.

⚙️ Accessibility Features

Platform-Wide Accessibility Checklist

Across every screen and workflow, these foundational features are in place.

Skip to Main Content

A skip navigation link is the first focusable element on every page, allowing keyboard users to bypass repeated navigation.

Descriptive Alt Text

All meaningful images carry concise, descriptive alt text. Purely decorative images use empty alt="" so screen readers skip them.

Logical Heading Hierarchy

Every page uses a single H1 followed by a logical hierarchy of H2–H6 headings, enabling screen reader users to navigate by heading structure.

Accessible Form Labels

All form inputs are associated with visible labels. Required fields and error messages are clearly identified to assistive technologies.

Reduced Motion Respect

All animations and transitions are disabled when the user's operating system is set to "prefer reduced motion", eliminating vestibular and seizure risks.

Responsive & Zoom-Safe

Content reflows correctly at 400% browser zoom with no loss of functionality or horizontal scrolling, supporting users with low vision.

Touch Target Sizing

All interactive touch targets are a minimum of 44×44 CSS pixels, meeting WCAG 2.5.5 Target Size for users with motor impairments.

Timeout Warnings

Session timeouts notify users at least 2 minutes in advance and allow an extension, ensuring users who need more time are not unexpectedly logged out.

Our Standards Roadmap

FinPay tracks its accessibility journey transparently. Here's where we stand and where we're heading.

All Level A success criteria met across our core payment flows and dashboard. Verified by a third-party accessibility auditor.
